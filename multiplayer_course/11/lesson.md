# Урок 11. Магия сетевых декораторов

## Введение

Вам когда-нибудь казалось, что мультиплеерный код выглядит немного перегруженным? Постоянные циклы `for msg in self.ctx.poll():`, куча условий `if msg.get("event") == ...` и ручная работа с `ctx.send_every()`. 

В этом уроке мы разберем **Сетевые Декораторы**. Они позволяют спрятать всю грязную работу под капот SpritePro, делая написание мультиплеерных игр быстрым и приятным, точно так же, как это делается во взрослых движках вроде Unity (с фреймворком Mirror).

## 1. Декоратор @s.NetEvent

Начнем с самого простого. Часто вам нужно просто слушать какое-то событие и реагировать на него. 
Вместо того чтобы проверять очередь сообщений руками, мы можем "пометить" функцию, чтобы она автоматически вызывалась при получении нужного сообщения.

```python
import spritePro as s

class Setup(s.Scene):
    def update(self, dt):
        # Отправляем сообщение на сервер
        if s.input.was_pressed(pygame.K_SPACE):
            self.ctx.send_every("ping", {"message": "Привет!"})

# SpritePro сам вызовет эту функцию, когда по сети придет событие "ping"!
@s.NetEvent("ping")
def on_ping_received(sender_id, message):
    print(f"Игрок с ID {sender_id} сказал: {message}")
```

**Плюсы**: Код избавляется от спагетти. Вся логика четко разнесена по событиям!

## 2. Разделение логики: @s.Command и @s.ClientRpc

С развитием игры вам потребуется строгая логика:
- **Хост (Сервер)** должен быть главным. Он проверяет правила (можем ли мы двигаться, есть ли мана для выстрела).
- **Клиент (Игрок)** должен отправлять запросы и ждать ответа от Хоста.

Для этого в SpritePro встроены два мощных декоратора: `@s.Command` и `@s.ClientRpc`.

### `@s.Command` — Запрос к Серверу
Вызывается Клиентом (или Хостом), но **исполняется ТОЛЬКО на Хосте**. 

Разберем на примере стрельбы: когда Клиент нажимает кнопку, он ПРОСИТ сервер создать пулю.
```python
@s.Command
def request_spawn_bullet(pos, direction, sender_id=None):
    # Этот код исполняется только на Хосте (даже если вызвал Клиент)
    # sender_id автоматически указывает, КТО сделал этот запрос.
    
    print(f"[Host] Игрок {sender_id} хочет выстрелить!")
    # Вызываем создание пули на экранах всех игроков
    spawn_bullet_rpc(pos=pos, direction=direction, owner=sender_id)
```

### `@s.ClientRpc` — Указание Клиентам
Вызывается Хостом, но **исполняется НА ВСЕХ Клиентах** (и на Хосте тоже, ведь Хост — тоже Клиент в игре).

```python
@s.ClientRpc
def spawn_bullet_rpc(pos, direction, owner):
    # Этот код сработает на всех компьютерах одновременно!
    bullet = s.Sprite("bullet.png", pos=pos)
    bullet.velocity = direction
```

Итоговая связка работает так:
1. `Клиент` нажимает SPACE -> `request_spawn_bullet()`. Ничего не рисуется, в сеть улетел пакет.
2. `Сервер` получает пакет и запускает `request_spawn_bullet()`. Смотрит логику и вызывает `spawn_bullet_rpc()`.
3. `Все клиенты` получают пакет и запускают `spawn_bullet_rpc()`. Появляется пуля!

## 3. Как узнать инфу обо всех игроках в Лобби?

Представьте, что мы делаем список игроков по клавише TAB. Нам нужно знать, кто вообще подключен, кто как себя назвал в лобби, и какие у них ID.

Обычно Лобби сохраняет никнеймы и рассылает их по сети. И SpritePro заботливо складывает эти данные в `self.ctx.players`!

Это словарь `dict`. 
- **Ключ** — это `client_id` (число)
- **Значение** — данные об игроке (тоже `dict`, обычно там есть ключ `"name"`).

Пример вывода списка всех игроков в лобби (например, при нажатии `TAB`):
```python
    def update(self, dt):
        if s.input.was_pressed(pygame.K_TAB):
            print("--- Текущие игроки в сессии ---")
            for player_id, data in self.ctx.players.items():
                print(f"ID {player_id}: Никнейм -> {data.get('name')}")
            print("-------------------------------")
```
Туда же вы можете вписывать свои роли, цвета и прочую информацию на этапе подключения через лобби, если немного изменить стандартное лобби SpritePro.
