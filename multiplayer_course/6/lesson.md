# Урок 6. Физика и игровые события: Зона захвата

В этом уроке мы разберем синхронизацию игрового счета и использование физических триггеров (зон) в сетевой игре.

## 1. Кто считает очки? (Authority)

В сетевых играх всегда должен быть «судья», который решает, было ли событие легитимным. 
- **Клиент**: Делает локальную проверку («Кажется, я вошел в зону») и шлет запрос серверу.
- **Хост (Сервер)**: Получает запрос, проверяет координаты игрока и, если всё верно, начисляет очко.

Если каждый клиент будет сам прибавлять себе очки, читеры смогут легко взломать игру. Поэтому **счет — это состояние, которое контролирует только Хост**.

## 2. Кулдауны и анти-спам

Сетевые пакеты могут дублироваться или приходить слишком часто. Чтобы одно касание зоны не превратилось в 100 очков за секунду, мы используем `cooldown`.

**Пример логики на клиенте:**
```python
score_cooldown = max(0.0, score_cooldown - s.dt)

# Проверка дистанции до центра (400, 300)
if score_cooldown <= 0.0 and dist_to_center < 30:
    ctx.send("score_request", {"id": my_id})
    score_cooldown = 1.0 # Ждем 1 секунду перед следующим начислением
```

## 3. Обработка на Хосте

Хост должен аккумулировать данные от всех и рассылать финальный результат.

**Пример кода (Host):**
```python
scores = {"host": 0, "client": 1}

for msg in ctx.poll():
    if msg.get("event") == "score_request" and ctx.is_host:
        pid = msg.get("data")["id"]
        scores[pid] += 1
        
        # Рассылаем ОБНОВЛЕННЫЙ счет всем (Broadcast)
        ctx.send("score_update", {"scores": scores})
```
