## 1. Проблема: «в двух окнах разные игры»

Типичная ошибка начинающих: каждый клиент сам симулирует еду, ботов и счёт. Через 10 секунд у хоста и клиента разные миры: еда в разных местах, боты «свои» у каждого окна, лидерборды не совпадают. Сообщения при этом честно ходят по сети — рассинхрон не в транспорте, а в **архитектуре**.

Причина: у объекта нет владельца. Оба окна считают себя «главными» по одной и той же еде.

## 2. Правило владения

**У каждого игрового объекта ровно один владелец. Остальные — зрители.**

| Объект | Владелец | Почему |
|--------|----------|--------|
| Змейка/персонаж игрока | сам игрок | свою позицию он знает точнее всех (client-side prediction, урок 10) |
| Еда, монеты, лут | хост | общий ресурс — нужен один арбитр |
| Боты, NPC | хост | иначе у каждого окна будут «свои» боты |
| Счёт | хост | клиент **запрашивает**, хост **решает** (урок 6) |
| Смерть игрока | сам игрок | свои столкновения он видит без задержки |

Владелец симулирует объект и рассылает его состояние. Остальные только рисуют то, что пришло, — без всякой логики. В слизарио для этого есть отдельный класс `NetSnakeView`: «змейка без мозгов», один и тот же для чужих игроков и для ботов на клиенте.

Проверка «я владелец общих объектов?» одной строкой, которая заодно делает сцену рабочей и в одиночном режиме:

```python
self.is_authority = ctx is None or ctx.is_host  # одиночная игра или хост
```

## 3. Два способа синхронизации: события и снапшоты

| | События (`coin_taken`) | Снапшот (`world`) |
|-|------------------------|-------------------|
| Что передаётся | что **изменилось** | **всё** состояние целиком |
| Трафик | минимальный | больше (но полный мир — единицы КБ) |
| Потерянное сообщение | рассинхрон **навсегда** | исправится следующим снапшотом |
| Подключение в середине игры | нужна спец-логика «передай новичку всё» | работает **само** |
| Код | сложнее (все переходы состояний) | тривиальный: сверил и починил |

Лучшая практика для небольших игр — **и то, и другое**:

- события — мгновенная реакция (монета исчезла сразу);
- периодический полный снапшот — сверка и самовосстановление (2 раза/с достаточно).

```python
# Владелец (хост): событие сразу + полный снапшот по таймеру
ctx.send("coin_taken", {"coin_id": cid, "taker": sender})
ctx.send_every("world", {"coins": [...], "scores": {...}}, 0.5)

# Клиент: сверка с полным снапшотом
def apply_snapshot(data):
    incoming = {int(c["id"]): c for c in data["coins"]}
    for cid in list(coins):          # исчезло у хоста — убираем
        if cid not in incoming:
            remove_coin(cid)
    for cid, c in incoming.items():  # появилось у хоста — добавляем
        if cid not in coins:
            make_coin(cid, tuple(c["pos"]))
```

Ключевая деталь — **у каждого объекта есть id**, который выдаёт владелец. Без id снапшот нельзя сверить с локальной копией (не поймёшь, «та же это монета или новая»), а события не к чему привязать.

### Подключение в середине игры — бесплатно

Раз всё состояние регулярно рассылается целиком, новому клиенту не нужен «пакет для новичка»: он просто начинает слушать и за полсекунды получает мир. В слизарио так синхронизируется всё: змейки — полное состояние 30 раз/с, боты — 20 раз/с, еда — снапшот каждые 0.5 с.

Когда игра вырастет (сотни объектов), снапшоты станут дорогими — тогда пригодятся дельты из урока 10. Но начинать всегда стоит с полных снапшотов: сложность дельт нужно ещё заслужить.

## 4. Оптимистичное действие + подтверждение владельца

Что делать, когда **клиент** трогает **чужой** объект (монету хоста)? Два наивных варианта плохи:

- клиент сам засчитывает — двое заберут одну монету;
- клиент ждёт разрешения хоста даже чтобы убрать монету с экрана — на пинге монета «липнет».

Правильный паттерн: **убирай сразу, засчитывай по подтверждению**.

```python
# Клиент: коснулся монеты
remove_coin(cid)                          # с экрана — мгновенно (отзывчивость)
ctx.send("take_coin", {"coin_id": cid})   # а вот счёт — только через хоста

# Хост: пришёл take_coin
if remove_coin(coin_id):                  # False, если её уже взял другой
    scores[sender] += 1
    ctx.send("coin_taken", {"coin_id": coin_id, "taker": sender})
```

Если двое взяли монету одновременно, хост подтвердит только первый запрос — второй игрок не получит очко, а его локально убранная монета всё равно исчезла бы. Конфликт разрешён без единой строчки специального кода.

### Гонка снапшота и события

Тонкий баг: хост собрал снапшот **до** того, как обработал ваш `take_coin`, — и снапшот «воскресит» монету, которую вы уже убрали. Лечится памятью о недавно удалённых id:

```python
recently_taken[cid] = time.monotonic()    # при удалении

# в apply_snapshot: не добавлять монету, взятую < 1 с назад
fresh = time.monotonic() - recently_taken.get(cid, -1.0) < 1.0
if cid not in coins and not fresh:
    make_coin(cid, pos)
```

Id при этом никогда не переиспользуются (владелец выдаёт `next_id += 1`), поэтому «карантин» старого id не заблокирует новую монету.

## 5. Смерти и отключения

- О своей смерти игрок сообщает сам (`player_died` + позиции сегментов) — хост превращает тело в еду и форсирует внеплановый снапшот.
- Отключение хост узнаёт из сервисного события `client_disconnected` (оно приходит только хосту из очереди сервера) и рассылает остальным `player_left`.
- Подстраховка от «молчаливых» вылетов — таймаут: нет пакетов от игрока 5 секунд → убираем его объект.

Три уровня защиты (событие → сервисное событие → таймаут) — стандартный набор для любой сетевой игры.
